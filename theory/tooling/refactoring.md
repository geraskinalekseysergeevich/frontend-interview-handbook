# Рефакторинг: чистый код без переписывания с нуля

Каждый разработчик рано или поздно открывает файл и думает: «Кто это написал?» Потом смотрит на git blame и видит собственное имя. Рефакторинг — это систематическая практика, которая помогает превращать такой код в читаемый и поддерживаемый — шаг за шагом, без риска всё сломать.

---

## Что такое рефакторинг

**Рефакторинг** — это процесс изменения внутренней структуры кода без изменения его внешнего поведения. Программа после рефакторинга делает ровно то же самое, что и до него — только код стал чище, понятнее и проще в поддержке.

Цели рефакторинга:
- Сделать код понятным для других (и для себя через полгода)
- Убрать дублирование
- Упростить добавление новой функциональности
- Снизить стоимость изменений в будущем

### Что НЕ является рефакторингом

Это важно понимать, чтобы не путать понятия:

- **Исправление багов** — если вы меняете поведение кода, это не рефакторинг, даже если меняете его в лучшую сторону.
- **Добавление новой функциональности** — новые фичи меняют поведение системы.
- **Переписывание с нуля** — замена одного решения другим с нуля. Рефакторинг — это серия мелких шагов, а не «давайте перепишем всё заново».
- **Оптимизация производительности** — оптимизация может делать код менее читаемым ради скорости. Это другая задача.

Чёткое разделение помогает: если вы рефакторите, вы не добавляете функциональность. Если добавляете — не рефакторите. Смешивать опасно.

## Когда рефакторить

### Правило бойскаута

«Оставляй лагерь чище, чем нашёл его». Применительно к коду: **каждый раз, когда вы открываете файл — улучшите хоть что-нибудь**. Не обязательно переписывать всё — переименуйте непонятную переменную, вынесите магическую константу, разбейте слишком длинную функцию.

Этот подход не требует выделять отдельного времени на рефакторинг — он становится частью ежедневной работы.

### Цикл «красный — зелёный — рефакторинг»

В TDD (разработке через тестирование) рефакторинг — обязательный третий шаг:

1. **Красный** — пишем тест, который описывает желаемое поведение. Тест падает, потому что функциональности ещё нет.
2. **Зелёный** — пишем минимальный код, чтобы тест прошёл. Не думаем о красоте — главное зелёный.
3. **Рефакторинг** — теперь, когда поведение зафиксировано тестом, улучшаем код. Тесты всё ещё зелёные — значит, мы ничего не сломали.

Этот цикл даёт уверенность: тесты — ваша страховка.

### Рефакторинг при добавлении фичи

Часто лучший момент для рефакторинга — перед тем, как добавить новую функциональность. Если код сложно расширять, сначала приведите его в порядок, а потом добавляйте фичу. Мартин Фаулер описывает это так: «Когда добавить функцию трудно, сначала сделай добавление лёгким, а потом добавь функцию».

## Code smells базового уровня

«Запах кода» — это характеристика кода, которая сигнализирует о потенциальной проблеме. Сам по себе запах — не баг, но намекает, что здесь стоит посмотреть внимательнее.

### Длинная функция

Функции, которые выросли до 50, 100, 200 строк — почти всегда делают слишком много. Их сложно читать, тестировать и переиспользовать.

```javascript
// Плохо: функция делает три разные вещи
function processOrder(order) {
  // 1. Валидация
  if (!order.items || order.items.length === 0) {
    throw new Error('Нет товаров');
  }
  if (!order.userId) {
    throw new Error('Нет пользователя');
  }

  // 2. Подсчёт стоимости
  let total = 0;
  for (const item of order.items) {
    total += item.price * item.quantity;
  }
  if (order.promoCode === 'SALE10') {
    total *= 0.9;
  }

  // 3. Сохранение
  db.orders.save({ ...order, total, status: 'pending' });
  emailService.send(order.userId, `Ваш заказ на ${total} руб. принят`);
}
```

### Длинный список параметров

Если функция принимает 4+ параметра — это сигнал. Функцию трудно вызывать, легко перепутать порядок аргументов.

```javascript
// Плохо
function createUser(firstName, lastName, email, age, role, isActive) { ... }

// Лучше: объект-параметр
function createUser({ firstName, lastName, email, age, role, isActive }) { ... }
```

### Дублирование кода

Если один и тот же блок логики встречается в двух и более местах — это проблема. При изменении нужно помнить обновить все копии. Рано или поздно о какой-то забудут.

```javascript
// Плохо: одинаковая логика в двух местах
function getAdminUsers(users) {
  return users.filter(u => u.role === 'admin' && u.isActive);
}

function getEditorUsers(users) {
  return users.filter(u => u.role === 'editor' && u.isActive);
}

// Хорошо: параметризованная функция
function getUsersByRole(users, role) {
  return users.filter(u => u.role === role && u.isActive);
}
```

### Магические числа и строки

Числа и строки без объяснения их смысла делают код нечитаемым. Что значит `86400`? А `7`? А `'admin'`?

```javascript
// Плохо: непонятные числа
if (user.sessionAge > 86400) {
  logout();
}

// Хорошо: константа с именем
const SESSION_EXPIRY_SECONDS = 86400; // 24 часа

if (user.sessionAge > SESSION_EXPIRY_SECONDS) {
  logout();
}
```

### Глубокая вложенность

Код с четырьмя-пятью уровнями отступов читается как квест: чтобы понять, что происходит внутри, нужно держать в голове все условия снаружи.

```javascript
// Плохо: «стрелочный антипаттерн»
function processPayment(payment) {
  if (payment) {
    if (payment.amount > 0) {
      if (payment.currency === 'RUB') {
        if (payment.userId) {
          // основная логика — глубоко закопана
          return doPayment(payment);
        }
      }
    }
  }
}
```

## Техники рефакторинга

### Extract Function (Выделение функции)

Берём фрагмент кода, который можно объяснить отдельным именем, и выносим его в функцию.

```javascript
// До
function printInvoice(invoice) {
  console.log(`Счёт #${invoice.id}`);
  console.log(`Клиент: ${invoice.customer}`);

  // подсчёт итога
  let total = 0;
  for (const line of invoice.lines) {
    total += line.amount;
  }
  console.log(`Итого: ${total} руб.`);
}

// После
function printInvoice(invoice) {
  printHeader(invoice);
  console.log(`Итого: ${calculateTotal(invoice.lines)} руб.`);
}

function printHeader(invoice) {
  console.log(`Счёт #${invoice.id}`);
  console.log(`Клиент: ${invoice.customer}`);
}

function calculateTotal(lines) {
  return lines.reduce((sum, line) => sum + line.amount, 0);
}
```

### Rename Variable (Переименование переменной)

Имя переменной должно рассказывать, что в ней хранится. `d`, `tmp`, `data`, `x` — почти всегда плохие имена.

```javascript
// До
const d = new Date();
const elapsed = d - startTime;

// После
const currentDate = new Date();
const elapsedMilliseconds = currentDate - startTime;
```

### Replace Magic Number with Constant

Уже показали выше, повторим основное: каждое магическое число — кандидат на именованную константу.

```javascript
// До
const discount = price * 0.15;
setTimeout(refresh, 300000);

// После
const LOYALTY_DISCOUNT_RATE = 0.15;
const AUTO_REFRESH_INTERVAL_MS = 300_000; // 5 минут

const discount = price * LOYALTY_DISCOUNT_RATE;
setTimeout(refresh, AUTO_REFRESH_INTERVAL_MS);
```

### Early Return / Guard Clauses (Ранний возврат)

Вместо глубокой вложенности — проверяем «плохие» случаи в начале функции и выходим из неё. Основная логика остаётся на верхнем уровне отступа.

```javascript
// До: глубокая вложенность
function processPayment(payment) {
  if (payment) {
    if (payment.amount > 0) {
      if (payment.currency === 'RUB') {
        if (payment.userId) {
          return doPayment(payment);
        }
      }
    }
  }
  return null;
}

// После: guard clauses
function processPayment(payment) {
  if (!payment) return null;
  if (payment.amount <= 0) return null;
  if (payment.currency !== 'RUB') return null;
  if (!payment.userId) return null;

  return doPayment(payment); // основная логика — без лишних отступов
}
```

Читать такой код значительно проще: каждая строка — одна проверка, основной сценарий внизу без вложенности.

## Почему рефакторинг без тестов опасен

Рефакторинг по определению не меняет поведение. Но как вы убедитесь, что не поменяли, если нет тестов?

Без тестов рефакторинг — это угадывание. Вы переименовываете функцию, выносите фрагмент, меняете структуру — и надеетесь, что ничего не сломали. Ручное тестирование всех сценариев неэффективно и ненадёжно.

Тесты дают вам:
- **Уверенность**: прошли — значит, поведение сохранено.
- **Скорость**: не нужно вручную кликать по всем экранам после каждого изменения.
- **Смелость**: зная, что тесты поймают ошибку, вы рефакторите охотнее и глубже.

Если вы хотите отрефакторить код, но тестов нет — сначала напишите тесты на текущее поведение. Да, это дополнительная работа. Но это единственный безопасный путь.

```javascript
// Прежде чем рефакторить calculateDiscount — убеждаемся, что понимаем её поведение
describe('calculateDiscount', () => {
  it('возвращает 10% скидку для VIP-клиентов', () => {
    expect(calculateDiscount(1000, 'vip')).toBe(100);
  });

  it('возвращает 0 для обычных клиентов', () => {
    expect(calculateDiscount(1000, 'regular')).toBe(0);
  });
});
// Теперь можно смело менять внутреннюю реализацию
```
