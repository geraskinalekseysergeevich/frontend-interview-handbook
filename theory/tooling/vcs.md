# Git и системы контроля версий

Представьте, что вы пишете код, всё работает, а потом вы что-то меняете — и всё ломается. Без системы контроля версий единственный выход — нажать Ctrl+Z несколько десятков раз или вспоминать, что именно изменилось. Git решает эту проблему элегантно: он хранит полную историю всех изменений, позволяет откатиться в любой момент и работать в команде без конфликтов. Это не просто инструмент — это фундаментальный навык любого разработчика.

---

## Что такое VCS и зачем он нужен

**VCS (Version Control System)** — система, которая записывает изменения в файлы с течением времени. Грубо говоря, это машина времени для вашего кода.

Существует три поколения VCS:

**Локальные** — простейший вариант: копируете папку с проектом с датой в названии (`project_backup_2024-01-15`). Подвержены ошибкам, не помогают в командной работе.

**Централизованные (CVCS)** — единый сервер хранит всю историю. Разработчики получают только нужные версии файлов. Примеры: SVN, Perforce. Проблема: если сервер упал — работа стоит у всей команды.

**Распределённые (DVCS)** — каждый разработчик получает полную копию репозитория. Git относится именно к этой категории. Преимущества: можно работать офлайн, каждая копия — это полный бэкап, нет единой точки отказа.

Git создал Линус Торвальдс в 2005 году для разработки ядра Linux. Сегодня это де-факто стандарт в индустрии.

## Три состояния файлов

Git оперирует тремя принципиально разными состояниями файлов. Понимание этой модели — ключ к тому, чтобы Git перестал казаться магией.

```
┌─────────────────┐    git add    ┌──────────────┐    git commit    ┌────────────┐
│ Working Directory│ ────────────▶ │ Staging Area │ ───────────────▶ │ Repository │
│  (рабочая папка) │              │   (индекс)   │                  │ (история)  │
└─────────────────┘              └──────────────┘                  └────────────┘
```

**Working Directory** — это папка на вашем диске, где вы редактируете файлы. Здесь живут ваши изменения до того, как вы решите их зафиксировать.

**Staging Area** (индекс) — промежуточная зона. Вы явно отмечаете, какие изменения войдут в следующий коммит. Это позволяет коммитить не все изменения сразу, а только логически связанные части. Например, вы поправили баг и заодно отрефакторили соседний файл — можно сделать два отдельных коммита.

**Repository** — база данных Git, которая хранит историю всех коммитов. Находится в папке `.git` в корне проекта.

```bash
# Посмотреть текущее состояние файлов
git status

# Вывод покажет:
# - Changes not staged for commit (изменены, но не в staging)
# - Changes to be committed (в staging, готовы к коммиту)
# - Untracked files (Git их ещё не отслеживает)
```

## Базовые команды

### Инициализация и клонирование

```bash
# Инициализировать новый репозиторий в текущей папке
git init

# Клонировать существующий репозиторий
git clone https://github.com/user/repo.git

# Клонировать в конкретную папку
git clone https://github.com/user/repo.git my-project
```

### Работа с изменениями

```bash
# Добавить конкретный файл в staging area
git add index.js

# Добавить все изменения в текущей папке
git add .

# Добавить части файла интерактивно (удобно для точечных коммитов)
git add -p

# Зафиксировать изменения из staging area
git commit -m "feat: add user authentication"

# Добавить все отслеживаемые файлы и сразу закоммитить
git commit -am "fix: correct typo in header"
```

### Синхронизация с удалённым репозиторием

```bash
# Отправить изменения на удалённый сервер
git push origin main

# Получить изменения с удалённого сервера (fetch + merge)
git pull origin main

# Только скачать информацию об изменениях, не применяя их
git fetch origin

# Применить скачанные изменения
git merge origin/main
```

Разница между `pull` и `fetch`: `fetch` только скачивает данные, не меняя рабочие файлы. `pull` — это `fetch` + `merge` в одной команде. В командной работе часто предпочтительнее `fetch`, чтобы сначала посмотреть, что изменилось.

### Git Stash — временное хранилище

Stash — это как ящик, куда можно временно спрятать незаконченные изменения. Полезно, когда нужно срочно переключиться на другую задачу.

```bash
# Спрятать все изменения (tracked файлы)
git stash

# Спрятать с описанием
git stash push -m "WIP: user profile page"

# Посмотреть список спрятанного
git stash list
# stash@{0}: WIP on feature/auth: abc1234 fix: button color
# stash@{1}: WIP: user profile page

# Восстановить последний stash (stash остаётся в списке)
git stash apply

# Восстановить и удалить из списка
git stash pop

# Восстановить конкретный stash
git stash apply stash@{1}

# Удалить конкретный stash
git stash drop stash@{0}

# Очистить все stash
git stash clear
```

## Ветки

Ветки — одна из самых мощных фич Git. Ветка — это просто указатель на конкретный коммит. Создавать ветки в Git дёшево и быстро (в отличие от SVN, где это занимало минуты).

### Основные операции с ветками

```bash
# Посмотреть все локальные ветки
git branch

# Посмотреть все ветки, включая удалённые
git branch -a

# Создать новую ветку
git branch feature/user-auth

# Переключиться на ветку
git checkout feature/user-auth

# Создать ветку и сразу переключиться (удобный шорткат)
git checkout -b feature/user-auth

# Современный вариант (Git 2.23+)
git switch -c feature/user-auth

# Переименовать текущую ветку
git branch -m new-name

# Переименовать конкретную ветку
git branch -m old-name new-name

# Удалить ветку (безопасно — не удалит незаслитую)
git branch -d feature/user-auth

# Удалить принудительно
git branch -D feature/user-auth

# Удалить ветку на удалённом сервере
git push origin --delete feature/user-auth
```

### Слияние веток

```bash
# Слить ветку feature/user-auth в текущую ветку
git merge feature/user-auth

# Слить без fast-forward (создаёт явный merge commit)
git merge --no-ff feature/user-auth
```

## .gitignore

Файл `.gitignore` говорит Git, какие файлы и папки игнорировать. Он должен находиться в корне репозитория.

```gitignore
# Игнорировать конкретный файл
.env

# Игнорировать папку
node_modules/

# Игнорировать все файлы с расширением
*.log
*.tmp

# Игнорировать все файлы в любой вложенной папке dist
**/dist/

# Исключить из игнорирования (антипаттерн !)
!important.log

# Игнорировать всё в папке build, кроме одного файла
build/*
!build/README.md
```

Типичный `.gitignore` для JavaScript-проекта:

```gitignore
# Зависимости
node_modules/

# Переменные окружения (никогда не коммитить!)
.env
.env.local
.env.production

# Артефакты сборки
dist/
build/
.cache/

# Логи
*.log
npm-debug.log*

# OS-файлы
.DS_Store
Thumbs.db

# IDE
.idea/
.vscode/
*.swp
```

Важно: если файл уже был закоммичен, добавление его в `.gitignore` не поможет. Нужно сначала удалить его из отслеживания:

```bash
git rm --cached .env
```

## Git Config

Перед первым коммитом нужно представиться Git — эта информация будет записана в каждый коммит.

```bash
# Установить имя и email глобально (для всех репозиториев)
git config --global user.name "Ivan Petrov"
git config --global user.email "ivan@example.com"

# Установить редактор для коммит-сообщений
git config --global core.editor "code --wait"  # VS Code
git config --global core.editor "vim"

# Установить настройку только для текущего репозитория
git config user.name "Work Account"

# Посмотреть все настройки
git config --list

# Посмотреть конкретную настройку
git config user.name
```

Конфиг хранится в `~/.gitconfig` (глобальный) и `.git/config` (локальный для репозитория).

## Merge Requests и Pull Requests

**Pull Request (PR)** в GitHub/GitLab называется **Merge Request (MR)** — суть одна. Это механизм предложить изменения из одной ветки в другую через веб-интерфейс, с возможностью обсуждения и code review.

Типичный workflow:
1. Создаёте ветку от `main`: `git checkout -b feature/add-search`
2. Разрабатываете, коммитите
3. Пушите ветку: `git push origin feature/add-search`
4. Открываете PR/MR через веб-интерфейс GitHub/GitLab
5. Коллеги делают code review, оставляют комментарии
6. Вы вносите правки и пушите новые коммиты
7. После одобрения ветка сливается в `main`

PR/MR важны не только для code review, но и как документация: история PR показывает, зачем и почему были сделаны изменения.

## Навигация по истории

```bash
# Посмотреть историю коммитов
git log

# Компактный вывод в одну строку
git log --oneline

# Красивый граф со всеми ветками
git log --oneline --graph --all

# История конкретного файла
git log --oneline -- src/utils.js

# Показать изменения в последнем коммите
git show

# Показать конкретный коммит
git show abc1234

# Сравнить рабочую директорию с последним коммитом
git diff

# Сравнить staging area с последним коммитом
git diff --staged

# Сравнить две ветки
git diff main feature/user-auth

# Узнать, кто написал каждую строку файла
git blame src/utils.js

# Показать, кто и когда изменил строки 10-20
git blame -L 10,20 src/utils.js
```

## Разрешение конфликтов

Конфликт возникает, когда два разработчика изменили одну и ту же строку в разных ветках, и Git не может автоматически решить, какая версия правильная.

```bash
# Пытаемся слить ветку
git merge feature/user-auth
# CONFLICT (content): Merge conflict in src/auth.js
# Automatic merge failed; fix conflicts and then commit the result.
```

Git добавляет маркеры в файл с конфликтом:

```javascript
<<<<<<< HEAD
// Ваша версия (текущая ветка)
function login(email, password) {
  return api.post('/auth/login', { email, password });
}
=======
// Версия из сливаемой ветки
function login(credentials) {
  return api.post('/auth/login', credentials);
}
>>>>>>> feature/user-auth
```

Нужно вручную отредактировать файл, оставив нужный код (убрав маркеры), затем:

```bash
# После разрешения конфликтов
git add src/auth.js
git commit -m "merge: resolve conflict in auth.js"
```

Советы по минимизации конфликтов:
- Регулярно синхронизируйтесь с основной веткой: `git pull origin main`
- Делайте ветки короткоживущими (1-3 дня)
- Разбивайте задачи так, чтобы разработчики работали в разных частях кода
