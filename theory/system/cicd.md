# CI/CD: автоматизация сборки и деплоя

Представьте: каждый разработчик в команде деплоит вручную. Один забыл запустить тесты, другой задеплоил на прод не ту ветку, третий обновил зависимости и не предупредил остальных. CI/CD решает именно эту проблему — автоматизирует всё, что команда должна делать одинаково при каждом изменении кода.

---

## Что такое CI (Continuous Integration)

**Непрерывная интеграция** — практика, при которой каждый разработчик регулярно (хотя бы раз в день) сливает свои изменения в общую ветку, а каждый такой слияние автоматически проверяется системой.

**Цель CI:** как можно раньше обнаружить проблему — до того, как она добралась до продакшена.

Как работает на практике:
1. Разработчик пушит ветку с изменениями.
2. CI-система автоматически запускает проверки: линтер, тесты, сборку.
3. Если что-то упало — разработчик получает уведомление и исправляет до мержа.
4. Если всё прошло — пуллреквест можно мержить.

Без CI такие ошибки всплывают позже: на код-ревью, после мержа, или — в худшем случае — в продакшене.

---

## Что такое CD: Delivery vs Deployment

За аббревиатурой CD стоят два похожих, но разных понятия:

**Continuous Delivery (Непрерывная доставка)** — автоматически подготавливает готовый к деплою артефакт (собранное приложение), но **финальный деплой на продакшен запускается вручную** — нажатием кнопки. Команда всегда готова задеплоить, но решение принимают люди.

**Continuous Deployment (Непрерывное развёртывание)** — идёт дальше: если все проверки прошли, код **автоматически уходит на продакшен** без участия человека. Каждый мерж в main — это деплой.

```
CI → CD (Delivery):    [push] → [lint+test+build] → [артефакт готов] → [деплой вручную]
CI → CD (Deployment):  [push] → [lint+test+build] → [автодеплой на прод]
```

Большинство команд используют Continuous Delivery: полный автоматизм пугает, и часто есть бизнес-причины выбирать момент деплоя вручную.

## Типичный CI-пайплайн

Пайплайн — это последовательность шагов (jobs), которые выполняются при каждом изменении кода. Классическая цепочка для фронтенд-проекта:

```
[push / pull request]
        ↓
   [1. Lint]         ← проверка стиля кода, ошибок TypeScript
        ↓
   [2. Tests]        ← unit-тесты, интеграционные тесты
        ↓
   [3. Build]        ← npm run build, проверяем что сборка не падает
        ↓
   [4. Deploy]       ← деплой на staging или production
```

Шаги выполняются **последовательно**: если линтер нашёл ошибки — тесты не запускаются. Это экономит время и ресурсы.

Каждый шаг имеет смысл:
- **Lint** — находит проблемы раньше тестов (быстрее).
- **Tests** — гарантируют, что поведение не сломалось.
- **Build** — убеждаемся, что приложение вообще собирается (бывает, что тесты зелёные, а сборка падает).
- **Deploy** — доставляем артефакт до окружения.

## GitHub Actions: как устроен workflow

GitHub Actions — самый популярный CI/CD инструмент для проектов на GitHub. Конфигурация хранится в файлах YAML в директории `.github/workflows/`.

**Основные понятия:**

- **Workflow** — весь процесс автоматизации, описанный в одном YAML-файле.
- **Event (триггер)** — событие, которое запускает workflow: `push`, `pull_request`, расписание `schedule`, ручной запуск `workflow_dispatch`.
- **Job** — набор шагов, выполняющихся на одном runner'е (виртуальной машине).
- **Step** — один шаг внутри job: запуск shell-команды или готового action.
- **Action** — переиспользуемый модуль. Например, `actions/checkout` клонирует репозиторий, `actions/setup-node` устанавливает Node.js.
- **Runner** — виртуальная машина, на которой выполняются jobs. GitHub предоставляет бесплатные runner'ы (Ubuntu, macOS, Windows).

**Пример реального workflow для React-приложения:**

```yaml
# .github/workflows/ci.yml

name: CI

# Триггеры: запускаем при пуше в main и при открытии/обновлении PR
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  # Job 1: проверка качества кода
  lint-and-test:
    runs-on: ubuntu-latest   # runner — Ubuntu

    steps:
      # Клонируем репозиторий
      - name: Checkout code
        uses: actions/checkout@v4

      # Устанавливаем Node.js нужной версии
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'         # кэшируем node_modules для скорости

      # Устанавливаем зависимости
      - name: Install dependencies
        run: npm ci            # npm ci — детерминированная установка из lock-файла

      # Запускаем линтер
      - name: Lint
        run: npm run lint

      # Запускаем тесты
      - name: Test
        run: npm run test -- --coverage

      # Собираем приложение
      - name: Build
        run: npm run build
```

Этот файл — единственное, что нужно добавить в проект, чтобы получить CI. GitHub автоматически обнаружит его и будет запускать при каждом пуше.

**Параллельные jobs** — можно запускать несколько независимых задач одновременно:

```yaml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps: [...]

  test:
    runs-on: ubuntu-latest
    steps: [...]

  # deploy выполнится только после успеха lint и test
  deploy:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps: [...]
```

## Зачем CI/CD нужен команде

Без автоматизации каждый разработчик надеется, что другие помнят запустить тесты перед мержем. С CI/CD — система не даёт смержить код с красными тестами физически.

Главные преимущества:
- **Ранняя обратная связь** — ошибку находят через 3 минуты после пуша, а не через 3 дня.
- **Единые стандарты** — линтер запускается одинаково у всех, не зависит от настроек редактора.
- **Уверенность в релизе** — если пайплайн зелёный, команда знает, что базовые проверки пройдены.
- **Скорость** — убрать ручные шаги = деплоить чаще и быстрее.
