# Docker: контейнеры для разработчика

«У меня работает» — классическая фраза, которая означает, что на машине разработчика одна версия Node.js, а на сервере другая. Или не хватает какой-то системной библиотеки. Или переменные окружения настроены иначе. Docker решает именно эту проблему: приложение запускается в изолированной среде, которая одинакова везде — на ноутбуке, в CI и в продакшене.

---

## Что такое контейнер и зачем Docker

**Контейнер** — это изолированный процесс, который запускается со своей файловой системой, переменными окружения и сетевыми настройками. Внутри контейнера — ровно то, что описано в его конфигурации: определённая версия Node.js, нужные зависимости, ваш код. Ничего лишнего с хост-машины туда не просачивается.

Контейнер легче виртуальной машины: он не эмулирует отдельное железо, а использует ядро операционной системы хоста. Запуск занимает секунды, а не минуты.

**Зачем Docker фронтендеру:**
- Запустить локально бэкенд-сервис или базу данных одной командой, без установки.
- Убедиться, что сборка работает в той же среде, что и в CI.
- Передать коллеге воспроизводимую среду разработки без инструкций «у тебя должна быть версия X».

---

## Образ vs контейнер

Это самая важная концептуальная разница в Docker.

**Образ (Image)** — шаблон, неизменяемый снимок файловой системы. Образ описывает, как должна выглядеть среда: какая ОС, какие программы, какой код. Образ нельзя «запустить» — он только для чтения.

Аналогия: образ — это рецепт блюда.

**Контейнер (Container)** — запущенный экземпляр образа. Из одного образа можно создать сколько угодно контейнеров. Каждый контейнер изолирован от других и от хост-системы.

Аналогия: контейнер — это готовое блюдо, приготовленное по рецепту.

```
Образ (nginx:alpine)
    ├── Контейнер 1 → отдаёт сайт на порту 8080
    ├── Контейнер 2 → отдаёт API-документацию на порту 8081
    └── Контейнер 3 → (остановлен)
```

## Базовые команды CLI

### Работа с образами

```bash
# Скачать образ с Docker Hub (реестр публичных образов)
docker pull nginx:alpine
docker pull node:20-alpine

# Список всех локальных образов
docker images

# Удалить образ
docker rmi nginx:alpine
```

### Работа с контейнерами

```bash
# Запустить контейнер из образа
docker run nginx:alpine

# Запустить в фоне (-d = detached)
docker run -d nginx:alpine

# Запустить с маппингом порта: порт хоста:порт контейнера
docker run -d -p 8080:80 nginx:alpine
# Теперь http://localhost:8080 → порт 80 внутри контейнера

# Запустить с переменными окружения
docker run -d -e NODE_ENV=production -e PORT=3000 my-app

# Запустить с монтированием директории: путь_хоста:путь_в_контейнере
docker run -d -v /home/user/app:/app node:20-alpine

# Список запущенных контейнеров
docker ps

# Список всех контейнеров (включая остановленные)
docker ps -a

# Остановить контейнер
docker stop <container_id>

# Удалить остановленный контейнер
docker rm <container_id>

# Остановить и сразу удалить
docker rm -f <container_id>

# Посмотреть логи контейнера
docker logs <container_id>
docker logs -f <container_id>   # -f = follow, в реальном времени

# Выполнить команду внутри запущенного контейнера
docker exec -it <container_id> sh   # открыть shell внутри контейнера
```

## Dockerfile: сборка своего образа

**Dockerfile** — текстовый файл с набором инструкций для создания образа. Docker читает его сверху вниз и строит образ послойно: каждая инструкция — новый слой.

### Основные инструкции

**`FROM`** — базовый образ, от которого наследуемся. Всегда первая инструкция.

```dockerfile
FROM node:20-alpine   # Alpine Linux + Node.js 20 — лёгкий базовый образ
```

**`WORKDIR`** — рабочая директория внутри контейнера. Все последующие команды выполняются в ней.

```dockerfile
WORKDIR /app   # создаёт директорию /app и переходит в неё
```

**`COPY`** — копирует файлы с хост-машины в образ.

```dockerfile
COPY package*.json ./      # сначала копируем только package.json и lock-файл
COPY . .                   # потом — весь исходный код
```

**`RUN`** — выполняет команду во время сборки образа. Результат фиксируется в новом слое.

```dockerfile
RUN npm ci --only=production   # устанавливаем зависимости
RUN npm run build              # собираем приложение
```

**`EXPOSE`** — документирует, на каком порту слушает приложение. Сам по себе порт не пробрасывает — это делает флаг `-p` при `docker run`.

```dockerfile
EXPOSE 3000
```

**`CMD`** — команда по умолчанию, которая выполняется при запуске контейнера. В отличие от `RUN`, выполняется не при сборке, а при старте.

```dockerfile
CMD ["node", "server.js"]
```

### Полный пример: Node.js приложение

```dockerfile
# Используем Node.js 20 на Alpine (лёгкий дистрибутив Linux)
FROM node:20-alpine

# Рабочая директория внутри контейнера
WORKDIR /app

# Копируем файлы зависимостей отдельно —
# это позволяет Docker кэшировать слой с node_modules
# и не переустанавливать их при каждом изменении кода
COPY package.json package-lock.json ./

# Устанавливаем только production-зависимости
RUN npm ci --only=production

# Копируем исходный код
COPY . .

# Собираем TypeScript / bundler
RUN npm run build

# Документируем порт приложения
EXPOSE 3000

# Команда запуска при старте контейнера
CMD ["node", "dist/server.js"]
```

### Пример для фронтенд-приложения с Nginx

```dockerfile
# Этап 1: сборка React/Vue-приложения
FROM node:20-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Этап 2: раздача статики через Nginx
FROM nginx:alpine

# Копируем собранные файлы из предыдущего этапа
COPY --from=builder /app/dist /usr/share/nginx/html

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

Это называется **multi-stage build**: итоговый образ содержит только Nginx и статические файлы, без Node.js и исходного кода — маленький и безопасный.

## docker build и docker run с флагами

### Сборка образа

```bash
# Собрать образ из Dockerfile в текущей директории
# -t — тег (имя:версия)
docker build -t my-app:1.0 .

# Указать другой Dockerfile
docker build -f Dockerfile.prod -t my-app:prod .

# Посмотреть созданные образы
docker images
```

### Запуск с флагами

```bash
# Все ключевые флаги вместе
docker run \
  -d \                          # фоновый режим (detached)
  -p 3000:3000 \                # маппинг порта хост:контейнер
  -v $(pwd)/data:/app/data \    # монтирование директории
  -e NODE_ENV=production \      # переменная окружения
  -e DATABASE_URL=postgres://...\
  --name my-running-app \       # имя контейнера (удобнее чем ID)
  my-app:1.0                    # образ
```

Флаги, которые используются чаще всего:

| Флаг | Описание |
|---|---|
| `-d` | Запустить в фоне |
| `-p <host>:<container>` | Пробросить порт |
| `-v <host>:<container>` | Смонтировать директорию |
| `-e KEY=VALUE` | Установить переменную окружения |
| `--name <name>` | Дать контейнеру имя |
| `--rm` | Удалить контейнер после остановки |
| `-it` | Интерактивный режим (для shell) |

```bash
# Типичный сценарий: запустить локально PostgreSQL без установки
docker run -d \
  --name postgres-local \
  -p 5432:5432 \
  -e POSTGRES_PASSWORD=mysecretpassword \
  -e POSTGRES_DB=myapp \
  postgres:16-alpine
```

Это запустит полноценный PostgreSQL за 5 секунд. После работы: `docker stop postgres-local && docker rm postgres-local`.
