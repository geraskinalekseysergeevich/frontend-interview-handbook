# Мониторинг и логирование

Написать код — это половина дела. Гораздо важнее знать, что происходит с этим кодом в продакшене: падает ли он, медленно ли работает, какие ошибки получают пользователи. Именно для этого существует мониторинг.

---

## Что такое мониторинг и зачем он нужен разработчику

Мониторинг — это наблюдение за состоянием работающего приложения в реальном времени. Без него вы узнаёте об ошибках от пользователей или случайно. С ним — система сама сообщает вам, что что-то пошло не так.

Фронтендеру мониторинг нужен чтобы:
- Узнавать об ошибках JavaScript в продакшене (там нет DevTools).
- Видеть, какие страницы медленные и у каких пользователей.
- Отлавливать регрессии после деплоя: «после вчерашнего релиза количество ошибок выросло в 5 раз».
- Понимать реальный опыт пользователей, а не только тестовые сценарии.

---

## Чтение логов: структура и уровни

Лог (журнал) — это хронологическая запись событий в системе. В браузере логи пишутся в консоль, на сервере — в файл или в специализированную систему (ELK, CloudWatch, Loki).

### Структура log-записи

Хорошо структурированная запись лога содержит:
- **Время** (timestamp) — когда произошло событие.
- **Уровень** (level) — насколько это важно.
- **Сообщение** — что произошло.
- **Контекст** — дополнительные данные: userId, requestId, URL.

```
2024-11-15T10:23:41Z [ERROR] Failed to load user profile { userId: 42, statusCode: 503 }
2024-11-15T10:23:42Z [WARN]  Retry attempt 1/3 for userId: 42
2024-11-15T10:23:43Z [INFO]  User profile loaded successfully { userId: 42, duration: 1243ms }
```

### Уровни логирования

| Уровень | Когда использовать |
|---|---|
| `DEBUG` | Детальная отладочная информация для разработки. В продакшене обычно выключена. |
| `INFO` | Нормальная работа системы: пользователь вошёл, запрос обработан. |
| `WARN` | Что-то подозрительное, но не критичное: медленный запрос, deprecated API. |
| `ERROR` | Ошибка, которая мешает работе: упал запрос, не удалось сохранить данные. |

```javascript
// Браузерная консоль — четыре уровня
console.debug('Данные компонента:', { props, state }); // только в dev
console.info('Пользователь авторизован:', userId);
console.warn('Используется устаревший метод getUser(), перейдите на fetchUser()');
console.error('Не удалось загрузить профиль:', error);
```

Уровни важны для фильтрации: в продакшене обычно собирают только `WARN` и `ERROR`, чтобы не утопить в шуме.

## Поиск ошибки по стектрейсу

Стектрейс (stack trace) — это цепочка вызовов функций, которая привела к ошибке. Читать его нужно **снизу вверх**: нижние строки — откуда всё началось, верхние — где именно упало.

```
TypeError: Cannot read properties of undefined (reading 'name')
    at UserCard (UserCard.tsx:23:18)          ← здесь упало
    at renderWithHooks (react-dom.js:14985)
    at mountIndeterminateComponent (react-dom.js:17811)
    at beginWork (react-dom.js:19049)
    at App (App.tsx:47:5)                     ← здесь был вызван UserCard
    at main (index.tsx:12:3)                 ← точка входа
```

**Что смотреть в первую очередь:**

1. **Первая строка** — тип ошибки и её описание. `TypeError: Cannot read properties of undefined` — обращение к свойству `undefined`.
2. **Первая строка вашего кода** в трейсе — `UserCard.tsx:23:18`. Строка 23, символ 18. Это место, где упало.
3. **Контекст выше** — кто вызвал этот компонент и с какими данными.

```typescript
// UserCard.tsx, строка 23 — вот проблема:
// user может быть undefined, если данные ещё не загрузились
const name = user.name; // ← TypeError

// Исправление:
const name = user?.name ?? 'Гость';
```

**Минифицированный стектрейс в продакшене** — часто выглядит нечитаемо:

```
at t (main.a3f91b.js:1:4832)
```

Для восстановления читаемых имён нужны **source maps** — файлы, которые связывают минифицированный код с исходным. Sentry и другие инструменты загружают source maps автоматически.

## Инструменты: браузерная консоль и Sentry

### Браузерная консоль

Основной инструмент при разработке. Помимо вывода сообщений, консоль умеет:

```javascript
// Таблица для массивов объектов — удобнее, чем console.log
console.table(users);

// Группировка — для структурированного вывода
console.group('Загрузка пользователя');
console.log('Запрос:', url);
console.log('Ответ:', data);
console.groupEnd();

// Замер времени
console.time('renderList');
renderList(items);
console.timeEnd('renderList'); // renderList: 12.3ms

// Стектрейс вручную — полезно при отладке асинхронного кода
console.trace('Откуда вызывается эта функция?');
```

### Sentry

В продакшене DevTools нет. Sentry — это сервис, который перехватывает ошибки JavaScript прямо в браузере пользователя и отправляет их на централизованную платформу.

**Как работает:**
1. Вы устанавливаете SDK в приложение (`@sentry/react`, `@sentry/vue` и т.д.).
2. SDK перехватывает необработанные ошибки через `window.onerror` и `unhandledrejection`.
3. Ошибка с контекстом (стектрейс, браузер, OS, пользователь) отправляется на сервер Sentry.
4. Похожие ошибки группируются в **Issues** — вы видите одну запись вместо тысячи одинаковых.

```javascript
// Инициализация Sentry
import * as Sentry from '@sentry/react';

Sentry.init({
  dsn: 'https://xxx@sentry.io/yyy',
  environment: 'production',
  // source maps подгружаются автоматически если настроен интеграционный плагин
});

// Ручная отправка ошибки с контекстом
try {
  await loadUserProfile(userId);
} catch (error) {
  Sentry.captureException(error, {
    extra: { userId, page: 'profile' },
  });
}
```

В интерфейсе Sentry вы видите: сколько пользователей затронула ошибка, когда она впервые появилась, какой браузер, стектрейс с читаемыми именами (через source maps), хлебные крошки — что делал пользователь перед ошибкой.

## Алертинг

Алерт — это автоматическое уведомление, которое срабатывает при нарушении заданных условий: количество ошибок превысило порог, время ответа выросло, сервер перестал отвечать.

Зачем нужен: вы не можете постоянно смотреть на дашборды. Алертинг смотрит за вас и сообщает, когда что-то требует внимания.

Примеры алертов:
- Количество `ERROR`-событий за 5 минут превысило 100.
- Новая ошибка появилась в продакшене впервые.
- Количество затронутых пользователей выросло в 2 раза за час.

Sentry умеет отправлять алерты в Slack, Email, PagerDuty — команда узнаёт о проблеме сразу, а не когда пользователи напишут в поддержку.
