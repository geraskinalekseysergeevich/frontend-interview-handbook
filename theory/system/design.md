# Системное проектирование для фронтендера

Когда говорят «системное проектирование» — часто представляют что-то из мира бэкенда: базы данных, очереди, балансировщики. Но фронтендер тоже работает в системе. Понимание того, как устроено приложение на всех уровнях, позволяет принимать осознанные решения: почему запрос идёт на один адрес, а не другой; зачем здесь CDN; почему эта страница рендерится на сервере, а не в браузере.

---

## Что такое системное проектирование применительно к фронтенду

Системное проектирование в контексте фронтенда — это понимание и планирование того, **как части приложения взаимодействуют друг с другом**: браузер, сеть, серверы, базы данных, сторонние сервисы.

Для джуниора это означает:
- Уметь объяснить, что происходит от момента ввода URL до появления страницы.
- Понимать, зачем нужны CDN, кэш, API.
- Уметь читать схему архитектуры и задавать правильные вопросы.

Для мидла это уже:
- Выбор между SPA и SSR исходя из требований.
- Понимание, где и как кэшировать данные.
- Проектирование структуры компонентов с учётом масштабирования.

## Клиент-серверная архитектура

В основе большинства веб-приложений лежит простая модель: **клиент запрашивает — сервер отвечает**. Всё общение происходит по протоколу HTTP.

### HTTP-цикл запрос — ответ

Когда пользователь вводит адрес в браузер и нажимает Enter:

1. Браузер (клиент) формирует **HTTP-запрос**: метод, URL, заголовки.
2. Запрос уходит через сеть на сервер.
3. Сервер обрабатывает запрос и возвращает **HTTP-ответ**: статусный код, заголовки, тело.
4. Браузер получает ответ и отрисовывает страницу.

```
GET /products?category=phones HTTP/1.1
Host: example.com
Accept: text/html
Cookie: session=abc123
```

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 12432

<!doctype html>
<html>...страница...</html>
```

**Методы HTTP:**

| Метод | Назначение |
|---|---|
| GET | Получить ресурс (не меняет данные) |
| POST | Создать новый ресурс |
| PUT / PATCH | Обновить ресурс |
| DELETE | Удалить ресурс |

### Что такое REST API

REST (Representational State Transfer) — архитектурный стиль для построения API. Ключевые принципы:

- **Ресурсы** — всё именуется существительными: `/users`, `/products/42`, `/orders/17/items`.
- **HTTP-методы** определяют действие: `GET /users` — список, `POST /users` — создать, `DELETE /users/5` — удалить.
- **Stateless** — каждый запрос содержит всю нужную информацию. Сервер не хранит состояние сессии между запросами (токен авторизации передаётся в каждом запросе).

```javascript
// Фронтенд делает REST-запрос
const response = await fetch('/api/users/42', {
  method: 'GET',
  headers: {
    'Authorization': 'Bearer eyJhbGci...',
    'Content-Type': 'application/json',
  },
});

const user = await response.json();
```

## Основные компоненты типичного веб-приложения

### Браузер (клиент)

Это среда выполнения вашего кода. Браузер парсит HTML, строит DOM, выполняет JavaScript, отправляет запросы, хранит cookies и localStorage.

### CDN (Content Delivery Network)

Сеть серверов, распределённых географически. Когда пользователь запрашивает статический ресурс (JS-бандл, картинку, шрифт) — CDN отдаёт его с ближайшего сервера, а не с основного.

Зачем: **меньше задержка** (latency), разгрузка основного сервера, кэширование на уровне сети.

```
Пользователь в Новосибирске
    ↓
CDN-сервер в Новосибирске (кэш)  ← если ресурс есть в кэше
    или
CDN-сервер в Новосибирске → основной сервер в Москве  ← если кэша нет
```

### Web-сервер

Принимает входящие HTTP-запросы. Статические файлы (HTML, CSS, JS) отдаёт сам. Динамические запросы проксирует на API-сервер.

Популярные: **Nginx**, **Apache**. В Node.js-окружениях часто сам Node выступает как web-сервер.

### API-сервер (бэкенд)

Содержит бизнес-логику. Получает запросы, обращается к базе данных, формирует ответ. Фронтенд с ним общается через REST API или GraphQL.

### База данных

Хранит постоянные данные: пользователей, заказы, продукты. Бывают реляционные (PostgreSQL, MySQL — таблицы и SQL) и NoSQL (MongoDB, Redis — документы, ключ-значение).

### Кэш

Промежуточный слой для хранения часто запрашиваемых данных. Позволяет не ходить в базу при каждом запросе. Самый популярный инструмент — **Redis**.

```
Запрос к API → проверяем кэш Redis
    ↓ есть        ↓ нет
  отдаём        идём в БД → сохраняем в кэш → отдаём
```

### Общая схема

```
[Браузер] ←→ [CDN] ←→ [Nginx] ←→ [API-сервер] ←→ [Redis (кэш)]
                                        ↓
                                  [База данных]
                                        ↓
                              [Внешние сервисы (email, SMS)]
```

## Монолит vs микросервисы

**Монолит** — всё приложение в одном процессе: один репозиторий, один деплой, одна кодовая база. Просто запускать, просто дебажить, хорошо работает на старте.

**Микросервисы** — приложение разбито на независимые сервисы, каждый из которых делает что-то одно: сервис авторизации, сервис заказов, сервис нотификаций. Каждый деплоится отдельно, масштабируется отдельно.

Для фронтендера разница чувствуется в API: в монолите обычно один API, в микросервисах — несколько, либо API Gateway, который их объединяет.

| | Монолит | Микросервисы |
|---|---|---|
| Сложность | Низкая | Высокая |
| Масштабирование | Всё целиком | Каждый сервис отдельно |
| Деплой | Один | Много |
| Старт | Быстро | Сложно |

**Практический совет:** большинство стартапов начинают с монолита. Микросервисы появляются, когда монолит уже давит своим размером или нужно масштабировать отдельные части.

## SPA vs SSR

**SPA (Single Page Application)** — браузер загружает один HTML-файл (по сути пустой), а потом JavaScript строит всю страницу на клиенте. Переходы между «страницами» происходят без перезагрузки — JS меняет URL и перерисовывает компоненты.

```
Браузер → сервер: GET /
Сервер → браузер: index.html (почти пустой + <script>)
Браузер выполняет JS → JS запрашивает данные через API → рендерит UI
```

Плюсы: быстрые переходы после загрузки, богатый интерактивный UX.
Минусы: медленный первый экран (Time to Interactive), хуже SEO без дополнительных усилий.

**SSR (Server-Side Rendering)** — сервер генерирует полноценный HTML с данными и отдаёт его браузеру. Пользователь видит контент сразу.

```
Браузер → сервер: GET /products/42
Сервер → БД: SELECT * FROM products WHERE id=42
Сервер → браузер: готовый HTML с данными о товаре
```

Плюсы: быстрый первый экран, хорошо для SEO.
Минусы: каждый переход — новый запрос к серверу, нагрузка на сервер.

**SSG (Static Site Generation)** — компромисс: страницы генерируются заранее при деплое и отдаются как статические файлы.

## Как понять архитектуру проекта, на котором работаешь

Несколько практических вопросов, которые стоит задать (или найти ответы самостоятельно):

1. **Где живёт API?** — найдите базовый URL в конфигурации или переменных окружения (`REACT_APP_API_URL`, `.env`).

2. **Как деплоится приложение?** — посмотрите `package.json` (скрипт `build`), CI-конфиги (`.github/workflows`), Dockerfile.

3. **Как хранится состояние?** — Redux, Zustand, React Query? Это подскажет, откуда данные берутся и как кэшируются на фронте.

4. **Есть ли API Gateway или BFF?** — иногда фронтенд ходит не напрямую в несколько сервисов, а через Backend for Frontend (BFF) — промежуточный слой.

5. **Как устроена аутентификация?** — JWT-токены? Сессии? OAuth через стороннего провайдера?

```bash
# Быстрый способ найти базовый URL API в проекте
grep -r "API_URL\|baseURL\|BASE_URL" src/ --include="*.ts" --include="*.tsx" -l
```
