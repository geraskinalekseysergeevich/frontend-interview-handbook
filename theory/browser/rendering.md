# Браузерный рендеринг

Когда пользователь вводит адрес в строку браузера и нажимает Enter, начинается цепочка событий, которая заканчивается отрисованной страницей. Понимание этой цепочки — Critical Rendering Path — позволяет писать быстрые интерфейсы и уверенно отвечать на вопросы о производительности на собеседовании.

---

## Как браузер получает HTML и строит DOM

После DNS-резолвинга, TCP-рукопожатия и TLS-переговоров браузер отправляет HTTP GET и получает ответ. Первые 14 КБ — самые важные: именно в этот первый пакет сервер должен уложить критичный контент (это ограничение TCP slow start).

Полученные байты браузер превращает в DOM в несколько шагов:

1. **Bytes → Characters** — сырые байты декодируются в символы согласно указанной кодировке (UTF-8 и т.д.).
2. **Characters → Tokens** — парсер токенизирует поток по спецификации HTML5: открывающий тег, закрывающий тег, атрибут, текст.
3. **Tokens → Nodes** — из токенов формируются объекты-узлы.
4. **Nodes → DOM Tree** — узлы связываются в иерархическое дерево с учётом вложенности тегов.

Важная деталь: DOM строится **инкрементально**. Браузер не ждёт полной загрузки HTML — он начинает парсинг и рендеринг, как только приходят первые данные. Именно поэтому пользователь видит страницу постепенно.

**Preload Scanner** — скрытый механизм, который параллельно с основным парсером просматривает ещё не обработанный HTML в поисках ресурсов (`<link>`, `<script>`, `<img>`) и ставит их на загрузку заранее. Это существенно ускоряет общее время загрузки.

Браузер также автоматически исправляет ошибки: добавляет `<html>`, `<head>`, `<body>`, если они отсутствуют, закрывает незакрытые теги, вставляет `<tbody>` в таблицы.

```html
<!-- Браузер распарсит это корректно даже без <html> и <body> -->
<h1>Привет</h1>
<p>Страница работает</p>
```

## CSSOM: как браузер парсит CSS

Параллельно с построением DOM браузер обрабатывает CSS и строит **CSSOM (CSS Object Model)** — аналогичное дерево, но для стилей.

Ключевое отличие от DOM: CSSOM строится **не инкрементально**. Браузер должен получить и обработать весь CSS-файл целиком, прежде чем использовать его. Причина проста — правила могут переопределять друг друга, и применить стиль "частично" значит рискнуть применить неверный результат.

```css
/* Это правило... */
p { color: black; }

/* ...будет перекрыто этим. Браузер должен знать оба до рендеринга */
p { color: red; }
```

CSSOM строится каскадно: сначала применяются стили браузера (user agent stylesheet), затем пользовательские стили, от общих правил к специфичным. Вес (специфичность) каждого правила определяет, какое из них победит.

Несмотря на то, что CSSOM звучит пугающе, время его построения обычно меньше одного DNS-lookup. Проблема не в скорости построения, а в том, что **CSS блокирует рендеринг** — браузер не начнёт строить render tree, пока не готов CSSOM.

## Render Tree: объединение DOM и CSSOM

Когда оба дерева готовы, браузер строит **Render Tree** — рабочую модель того, что реально нужно отрисовать.

Render Tree ≠ DOM. Из него исключаются:
- Элементы `<head>`, `<script>`, `<meta>` — они не имеют визуального представления.
- Элементы с `display: none` — они полностью убираются из потока.
- Текстовые узлы с одними пробелами.

При этом `visibility: hidden` в render tree **присутствует** — элемент невидим, но занимает место.

```css
.hidden-display { display: none; }    /* НЕТ в render tree */
.hidden-visibility { visibility: hidden; } /* ЕСТЬ в render tree */
```

Каждый узел render tree содержит элемент DOM и его вычисленные стили (computed styles).

## Layout, Paint, Composite

После построения render tree браузер проходит оставшиеся этапы пайплайна.

### Layout (Reflow)

На этом этапе браузер вычисляет **точные размеры и позиции** каждого элемента в пикселях. Он проходит render tree сверху вниз, рассчитывая box model каждого узла: ширину, высоту, отступы, позицию.

Layout зависит от размера viewport. Поэтому правило:

```html
<meta name="viewport" content="width=device-width, initial-scale=1">
```

критически важно: без него браузер использует дефолтную ширину ~960px и потом масштабирует результат — получается два layout вместо одного.

Layout запускается повторно (reflow) при любом изменении геометрии: изменении размеров окна, изменении CSS-свойств типа `width`, `height`, `margin`, `padding`, добавлении/удалении DOM-элементов. Reflow — дорогостоящая операция, она пересчитывает расположение всего вокруг изменённого элемента.

```js
// Плохо: вызывает reflow на каждой итерации
for (let i = 0; i < 100; i++) {
  el.style.width = el.offsetWidth + 1 + 'px'; // чтение offsetWidth сбрасывает очередь
}

// Лучше: читаем один раз, пишем один раз
const width = el.offsetWidth;
for (let i = 0; i < 100; i++) {
  el.style.width = width + i + 'px';
}
```

### Paint (Rasterization)

После layout браузер знает, **что** нарисовать и **где**. Теперь он переводит это в пиксели: рисует текст, цвета, изображения, тени, границы. Этот процесс называется rasterization.

При первой загрузке перерисовывается весь экран. При последующих изменениях браузер умеет перерисовывать только изменившиеся области (dirty regions).

Для поддержания 60 FPS у браузера есть ровно **16.67 мс** на каждый кадр. Если JavaScript или paint занимают больше — возникают "прыжки" (jank).

### Composite

Сложные страницы разбиваются на **слои (layers)**. Каждый слой рисуется отдельно, а затем GPU объединяет их в финальное изображение — это и есть compositing.

Слои создаются для элементов с:
```css
opacity: 0.5;
transform: translateZ(0); /* или любой 3D-transform */
will-change: transform;
position: fixed;
```

Смысл слоёв в том, что изменение одного слоя (например, анимация `transform`) не требует перерисовки остальных. GPU просто сдвигает готовый слой. Именно поэтому анимации через `transform` и `opacity` работают плавно — они происходят на уровне composite, минуя layout и paint.

```css
/* Плохая анимация — вызывает layout + paint на каждом кадре */
.bad { transition: width 0.3s; }

/* Хорошая анимация — только composite, GPU-accelerated */
.good { transition: transform 0.3s; }
```

## Critical Rendering Path

**Critical Rendering Path (CRP)** — это минимальная последовательность шагов, которую браузер должен завершить, чтобы отрисовать первый контент:

```
HTML → DOM
CSS  → CSSOM
         ↓
      Render Tree
         ↓
       Layout
         ↓
       Paint
         ↓
     Composite
```

Длина CRP измеряется в количестве критических ресурсов (тех, которые блокируют первый рендер), их суммарном размере и количестве сетевых round-trip'ов.

Стратегии сокращения CRP:

- **Минимизировать критические ресурсы**: откладывать некритичные CSS и JS.
- **Уложить всё нужное в первые 14 КБ**: HTML + критичные инлайн-стили.
- **Убрать блокировку парсера**: использовать `async`/`defer` для скриптов.

```html
<!-- Оптимальный порядок подключения -->
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">

  <!-- Критичные стили — инлайн или маленький файл -->
  <style>
    body { margin: 0; font-family: sans-serif; }
    header { background: #333; color: white; }
  </style>

  <!-- Некритичные стили — с медиа-атрибутом или асинхронно -->
  <link rel="stylesheet" href="print.css" media="print">
</head>
<body>
  <!-- Контент -->

  <!-- JS в конце тела или с defer/async -->
  <script defer src="app.js"></script>
</body>
```

## Влияние JavaScript на рендеринг

JavaScript — **parser-blocking** ресурс по умолчанию. Когда парсер встречает `<script>` без атрибутов, он:
1. Останавливает разбор HTML.
2. Ждёт загрузки скрипта (если внешний).
3. Выполняет скрипт.
4. Продолжает парсинг HTML.

Причина: JS может изменить DOM или CSSOM через `document.write()`, поэтому браузер не может безопасно продолжить разбор.

Помимо этого, если перед скриптом есть незагруженный CSS — браузер подождёт и его, поскольку JS может читать computed styles.

```html
<!-- Блокирует парсинг на время загрузки и выполнения -->
<script src="heavy.js"></script>

<!-- Загружается параллельно, выполняется когда загрузился (порядок не гарантирован) -->
<script async src="analytics.js"></script>

<!-- Загружается параллельно, выполняется после парсинга HTML, в порядке подключения -->
<script defer src="app.js"></script>

<!-- Инлайн-скрипт блокирует парсинг всегда -->
<script>
  document.querySelector('h1').textContent = 'Изменено';
</script>
```

Если главный поток занят выполнением JavaScript, браузер не может обрабатывать пользовательские события (клики, скролл). Страница формально загружена, но не реагирует — это называется "long task". Chrome DevTools показывает задачи длиннее 50 мс как проблемные.

## setTimeout и setInterval

Оба метода позволяют планировать выполнение кода, но работают через **очередь задач (task queue)**, а не мгновенно.

```js
console.log('1');
setTimeout(() => console.log('2'), 0);
console.log('3');

// Вывод: 1, 3, 2
// setTimeout(..., 0) не выполняется немедленно — он встаёт в очередь
```

`setTimeout(fn, delay)` планирует **однократное** выполнение. Минимальная задержка в браузере — 4 мс (даже если указать 0), а реальная задержка зависит от загруженности главного потока.

`setInterval(fn, delay)` планирует **повторяющееся** выполнение с заданным интервалом. Важный нюанс: если выполнение коллбека занимает больше времени, чем интервал — вызовы начнут "наслаиваться". Для надёжного повтора лучше использовать рекурсивный `setTimeout`:

```js
// Проблема с setInterval: если fn занимает 200мс, а интервал 100мс — очередь растёт
setInterval(fn, 100);

// Лучше: следующий вызов планируется только после завершения предыдущего
function repeat() {
  fn();
  setTimeout(repeat, 100);
}
repeat();
```

Оба метода возвращают числовой идентификатор таймера. Для отмены используют `clearTimeout(id)` и `clearInterval(id)`.

```js
const timerId = setTimeout(() => {
  console.log('Это никогда не выполнится');
}, 1000);

clearTimeout(timerId);
```

Таймеры и Event Loop тесно связаны: `setTimeout` помещает коллбек в **макротask queue**. Перед каждым новым макротаском браузер сначала опустошает очередь **микротасков** (Promise-обработчики, `queueMicrotask`). После каждого макротаска браузер может отрисовать кадр — именно поэтому `setTimeout(..., 0)` часто используют, чтобы "дать браузеру перед рисовать".
