# Сеть и HTTP

Каждый раз, когда браузер открывает страницу, скачивает картинку или отправляет форму — он общается с сервером по протоколу HTTP. Понимание этого протокола, структуры URL и инструментов работы с сетью — обязательный минимум для любого фронтенд-разработчика.

---

## Структура URL

URL (Uniform Resource Locator) — это адрес ресурса в сети. Разберём по частям:

```
https://example.com:443/api/users?role=admin&page=2#results
  │         │       │       │           │            │
схема      хост   порт    путь        query      fragment
```

- **Схема (scheme)** — протокол: `http`, `https`, `ftp`, `mailto`. Определяет, как именно передаются данные.
- **Хост (host)** — доменное имя или IP-адрес сервера: `example.com`, `192.168.1.1`.
- **Порт (port)** — номер порта на сервере. По умолчанию: 80 для HTTP, 443 для HTTPS. Если порт стандартный, его можно не писать.
- **Путь (path)** — конкретный ресурс на сервере: `/api/users`, `/images/logo.png`.
- **Query string** — параметры запроса после `?`, разделённые `&`: `role=admin&page=2`. Передаются на сервер, доступны через `location.search`.
- **Fragment (якорь)** — часть после `#`: `#results`. Браузер прокручивает к элементу с таким `id`. Fragment НЕ отправляется на сервер — он живёт только в браузере.

```javascript
const url = new URL('https://example.com:443/api/users?role=admin#results');
console.log(url.hostname);  // 'example.com'
console.log(url.pathname);  // '/api/users'
console.log(url.search);    // '?role=admin'
console.log(url.hash);      // '#results'
```

## HTTP методы

HTTP метод говорит серверу, что именно клиент хочет сделать с ресурсом. Это не просто условность — семантика методов имеет значение для кэширования, безопасности и правильного поведения API.

### GET — получить данные

Запрашивает представление ресурса. Не изменяет данные на сервере. Тело запроса не используется — параметры передаются через URL.

- **Safe** (безопасный): не изменяет состояние сервера
- **Idempotent** (идемпотентный): повторный вызов даёт тот же результат
- **Cacheable**: браузер и прокси могут кэшировать ответ

```http
GET /api/users?page=1 HTTP/1.1
Host: example.com
```

### POST — создать / отправить данные

Отправляет данные для обработки. Обычно создаёт новый ресурс. Тело запроса содержит данные. Не идемпотентен — два одинаковых POST могут создать два ресурса.

```http
POST /api/users HTTP/1.1
Content-Type: application/json

{"name": "Alice", "email": "alice@example.com"}
```

### PUT — заменить ресурс целиком

Заменяет весь ресурс переданными данными. Если ресурса нет — создаёт его. Идемпотентен: повторный вызов с теми же данными даст тот же результат.

```http
PUT /api/users/42 HTTP/1.1
Content-Type: application/json

{"name": "Alice", "email": "alice@example.com", "role": "admin"}
```

### PATCH — изменить часть ресурса

Применяет частичные изменения. Отправляете только то, что нужно изменить. В отличие от PUT, не требует передавать весь объект.

```http
PATCH /api/users/42 HTTP/1.1
Content-Type: application/json

{"role": "admin"}
```

### DELETE — удалить ресурс

Удаляет указанный ресурс. Идемпотентен: повторный DELETE уже удалённого ресурса не приводит к новым последствиям.

```http
DELETE /api/users/42 HTTP/1.1
```

### Сравнительная таблица

| Метод  | Safe | Idempotent | Тело запроса | Типичный сценарий          |
|--------|------|-----------|-------------|---------------------------|
| GET    | да   | да        | нет         | Получить список / элемент |
| POST   | нет  | нет       | да          | Создать ресурс            |
| PUT    | нет  | да        | да          | Обновить ресурс целиком   |
| PATCH  | нет  | нет       | да          | Частичное обновление      |
| DELETE | нет  | да        | нет         | Удалить ресурс            |

## Коды ответов HTTP

Сервер всегда возвращает трёхзначный код ответа. Первая цифра — группа, остальные — конкретный код.

### 1xx — Информационные

Промежуточные ответы. На практике редко встречаются во фронтенд-разработке.

- **100 Continue** — сервер принял заголовки запроса, клиент может слать тело.

### 2xx — Успех

Запрос обработан успешно.

- **200 OK** — всё хорошо. GET вернул данные, POST выполнил действие.
- **201 Created** — ресурс создан (обычно ответ на POST). В заголовке `Location` может быть URL нового ресурса.
- **204 No Content** — запрос выполнен, тело ответа пустое. Типично для DELETE или PATCH без возврата данных.

### 3xx — Перенаправление

Клиент должен сделать ещё один запрос по другому адресу.

- **301 Moved Permanently** — ресурс переехал навсегда. Браузер и поисковики запоминают новый URL.
- **302 Found** — временный редирект. Браузер идёт по новому URL, но старый не забывает.
- **304 Not Modified** — ресурс не изменился с последнего запроса. Браузер использует кэшированную копию. Сервер проверяет заголовки `If-Modified-Since` или `If-None-Match`.

### 4xx — Ошибка клиента

Запрос некорректен — виноват клиент.

- **400 Bad Request** — сервер не понял запрос: неправильный синтаксис, невалидные данные.
- **401 Unauthorized** — не аутентифицирован. Название misleading: означает «нужно войти», а не «нет прав».
- **403 Forbidden** — аутентифицирован, но нет прав на этот ресурс. Сервер знает кто ты, но не пускает.
- **404 Not Found** — ресурс не найден. URL не существует.
- **422 Unprocessable Content** — запрос синтаксически верен, но семантически некорректен. Часто используется для ошибок валидации: поле обязательное, но пустое.
- **429 Too Many Requests** — превышен rate limit. Сервер говорит: «ты слишком часто стучишься». Обычно содержит заголовок `Retry-After`.

### 5xx — Ошибка сервера

Запрос корректен, но сервер не смог его обработать.

- **500 Internal Server Error** — что-то пошло не так на сервере. Общая ошибка, когда сервер не знает что конкретно.
- **502 Bad Gateway** — сервер-посредник (прокси, балансировщик) получил невалидный ответ от вышестоящего сервера.
- **503 Service Unavailable** — сервер временно недоступен: перегрузка или техническое обслуживание. Должен содержать `Retry-After`.

## REST API

REST (Representational State Transfer) — архитектурный стиль для построения API. Это не стандарт и не протокол, а набор принципов.

### Ключевые принципы

1. **Клиент-серверная архитектура** — клиент и сервер разделены, общаются через интерфейс.
2. **Stateless** — каждый запрос содержит всю нужную информацию. Сервер не хранит состояние сессии.
3. **Кэшируемость** — ответы могут быть кэшированы.
4. **Единообразный интерфейс** — предсказуемая структура URL и методов.

### Ресурсы и эндпоинты

В REST всё — это ресурс. Ресурс описывается существительным во множественном числе:

```
/users          — коллекция пользователей
/users/42       — конкретный пользователь
/users/42/posts — посты конкретного пользователя
```

Действия выражаются через HTTP-методы, а не через URL:

```
GET    /users       — список пользователей
POST   /users       — создать пользователя
GET    /users/42    — получить пользователя #42
PUT    /users/42    — обновить пользователя #42
DELETE /users/42    — удалить пользователя #42
```

Плохой пример (не REST-стиль):
```
GET /getUsers
POST /createUser
GET /deleteUser?id=42
```

## Fetch API

`fetch` — современный способ делать HTTP-запросы из браузера. Приходит на смену `XMLHttpRequest`, работает на промисах.

### Базовое использование

```javascript
// GET-запрос
const response = await fetch('https://api.example.com/users');
const users = await response.json();
console.log(users);
```

Fetch работает в два этапа:
1. Промис разрешается с объектом `Response`, как только пришли заголовки ответа.
2. Вы вызываете метод на `Response` для получения тела — он тоже возвращает промис.

### Опции запроса

```javascript
const response = await fetch('https://api.example.com/users', {
  method: 'POST',                          // GET по умолчанию
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer token123',
  },
  body: JSON.stringify({ name: 'Alice' }), // только для POST/PUT/PATCH
  credentials: 'include',                  // отправлять cookies
  mode: 'cors',                            // cors, same-origin, no-cors
  cache: 'no-cache',                       // управление кэшем
});
```

### Чтение ответа

Объект `Response` предоставляет несколько методов для чтения тела — каждый возвращает промис:

```javascript
response.json()        // парсит JSON
response.text()        // возвращает текст
response.blob()        // для файлов/изображений
response.arrayBuffer() // для бинарных данных
response.formData()    // для FormData
```

Важные свойства `Response`:

```javascript
response.status   // числовой код: 200, 404, 500...
response.ok       // true если status в диапазоне 200-299
response.headers  // объект Headers
```

### Обработка ошибок

Главная ловушка: `fetch` отклоняет промис (reject) только при **сетевой ошибке** — нет интернета, сервер недостижим. Ответы 404 и 500 считаются успешными с точки зрения промиса!

```javascript
// Неправильно — ошибки сервера не поймаем
const response = await fetch('/api/users');
const data = await response.json(); // упадёт, если сервер вернул 500 с HTML

// Правильно — проверяем response.ok
async function fetchUsers() {
  try {
    const response = await fetch('/api/users');

    if (!response.ok) {
      throw new Error(`HTTP ошибка: ${response.status}`);
    }

    return await response.json();
  } catch (error) {
    // Сюда попадём при сетевой ошибке ИЛИ если мы сами бросили ошибку
    console.error('Запрос не удался:', error.message);
    throw error;
  }
}
```

### Отправка файлов

```javascript
const formData = new FormData();
formData.append('avatar', fileInput.files[0]);
formData.append('username', 'alice');

const response = await fetch('/api/profile', {
  method: 'PUT',
  body: formData,  // Content-Type выставится автоматически
});
```

## DevTools: вкладка Network

Network tab в DevTools браузера — главный инструмент для отладки HTTP-запросов.

### Что показывает

- **Список запросов** — все HTTP-запросы страницы: JS, CSS, изображения, API-вызовы.
- **Колонки**: Name, Status, Type, Initiator (кто инициировал), Size, Time.
- **Фильтры**: можно отфильтровать по типу — Fetch/XHR, JS, CSS, Img, Doc, WS (WebSocket).

### Как читать детали запроса

При клике на запрос открывается панель с вкладками:

- **Headers** — заголовки запроса и ответа, метод, URL, статус.
- **Payload** — тело запроса (для POST/PUT): query params, body.
- **Response** — тело ответа.
- **Preview** — отформатированный просмотр JSON/HTML.
- **Timing** — детализация времени запроса.

### Waterfall (водопад)

Колонка Waterfall показывает временную шкалу для каждого запроса. Цвета обозначают фазы:

- **Queueing** (серый) — запрос в очереди, браузер ещё не отправил.
- **Stalled** (серый) — ожидание свободного TCP-соединения.
- **DNS Lookup** (зелёный) — разрешение доменного имени в IP.
- **Initial connection** (оранжевый) — установка TCP-соединения.
- **SSL** (оранжевый) — TLS handshake для HTTPS.
- **TTFB** (зелёный) — Time To First Byte, время до начала получения ответа. Показывает скорость сервера.
- **Content Download** (синий) — загрузка тела ответа.

По водопаду сразу видно узкие места: долгий DNS, медленный TTFB, параллельность запросов.

### Полезные приёмы

```
Disable cache         — при открытом DevTools отключает кэширование
Throttling            — эмулирует медленное соединение (3G, Slow 3G)
Preserve log          — сохраняет запросы при навигации
Replay XHR            — повторить запрос правой кнопкой мыши
Copy as cURL          — скопировать запрос в формате cURL для терминала
```
