# Производительность браузера: загрузка ресурсов, скрипты, изображения, шрифты

Производительность — это не про красивые цифры в Lighthouse. Это про то, насколько быстро пользователь видит что-то полезное на экране и может с этим взаимодействовать. Браузер делает огромную работу за вас, но только вы решаете, в каком порядке и каком количестве он будет её выполнять.

---

## Как браузер загружает ресурсы: порядок и приоритеты

Когда пользователь открывает страницу, браузер получает HTML и начинает его разбирать сверху вниз. По ходу разбора он натыкается на ссылки на CSS, скрипты, изображения и шрифты — и начинает их скачивать. Всё это происходит не хаотично: у браузера есть встроенная система приоритетов.

Порядок загрузки в общем случае такой:

1. **HTML** — первый и главный. Без него браузер вообще не знает, что загружать.
2. **CSS** из `<head>` — загружается с высоким приоритетом, потому что без CSS браузер не строит визуальное дерево (CSSOM) и не начинает рендеринг.
3. **Скрипты** в `<head>` без атрибутов — блокируют разбор HTML (об этом ниже).
4. **Шрифты** — загружаются тогда, когда браузер понял, что они нужны, то есть после обработки CSS.
5. **Изображения** — обычно с низким приоритетом, если они не в области видимости.

Ресурсы выше в документе и в критическом пути рендеринга получают более высокий приоритет. Изображения ниже фолда браузер может отложить вовсе.

**Подсказка браузеру через `<link rel="preload">`:**

```html
<!-- Говорим браузеру: загрузи этот шрифт заранее, он понадобится -->
<link rel="preload" href="/fonts/Inter.woff2" as="font" type="font/woff2" crossorigin>

<!-- Заранее загружаем критический CSS -->
<link rel="preload" href="/css/critical.css" as="style">
```

`preload` не меняет порядок применения ресурса, но поднимает его приоритет загрузки. Это особенно полезно для шрифтов и LCP-изображений (Largest Contentful Paint).

## Влияние скриптов на загрузку: blocking, defer, async

Это одна из самых частых тем на собеседованиях — и одна из самых важных на практике.

По умолчанию `<script src="...">` без атрибутов **блокирует разбор HTML**. Браузер останавливается, скачивает скрипт, выполняет его, и только потом продолжает читать документ. Это называется **parser blocking** или **render blocking**.

```html
<!-- Плохо: браузер встанет здесь и будет ждать загрузки скрипта -->
<head>
  <script src="analytics.js"></script>
</head>
```

Почему так сделано? Потому что скрипт может вызвать `document.write()` или изменить структуру DOM, которую браузер ещё не дочитал.

### Три режима загрузки скриптов

```
Обычный:  [разбор HTML]→[стоп]→[загрузка]→[выполнение]→[разбор HTML продолжается]
async:    [разбор HTML + загрузка параллельно]→[стоп]→[выполнение]→[разбор HTML продолжается]
defer:    [разбор HTML + загрузка параллельно]→[HTML разобран]→[выполнение]→[DOMContentLoaded]
```

**`async`** — загружает скрипт параллельно с разбором HTML, но выполняет сразу, как только загрузится. Порядок выполнения нескольких `async`-скриптов **не гарантирован** — кто загрузился первым, тот и выполнился первым.

```html
<!-- Оба скрипта загружаются параллельно, но выполнятся в произвольном порядке -->
<script async src="counter.js"></script>
<script async src="widget.js"></script>
```

Когда использовать `async`: независимые скрипты, которым не нужен DOM и которые не зависят друг от друга. Хороший пример — счётчики аналитики типа Google Analytics.

**`defer`** — тоже загружает параллельно, но **откладывает выполнение** до момента, когда HTML полностью разобран. При этом несколько `defer`-скриптов выполняются **в том порядке, в котором написаны** в документе.

```html
<!-- jQuery загрузится параллельно, но выполнится после разбора HTML и до DOMContentLoaded -->
<script defer src="jquery.js"></script>
<!-- plugin.js гарантированно выполнится после jquery.js -->
<script defer src="plugin.js"></script>
```

Когда использовать `defer`: основная логика приложения, скрипты, которые работают с DOM, скрипты с зависимостями между собой.

**Итоговое правило:**

| Ситуация | Атрибут |
|---|---|
| Независимый скрипт (аналитика) | `async` |
| Скрипт работает с DOM | `defer` |
| Несколько взаимозависимых скриптов | `defer` для всех |
| Критический скрипт (must-run до рендера) | без атрибута, в `<head>` — но минимизировать |

**Место скрипта в документе тоже важно.** Классический паттерн — размещать скрипты перед `</body>`. Это работает, но сегодня предпочтительнее `defer` в `<head>`: браузер начинает загрузку раньше, а выполнение откладывает так же.

## Оптимизация изображений

Изображения — обычно самые тяжёлые ресурсы на странице. Правильный выбор формата и сжатие могут сократить вес страницы в несколько раз.

### Форматы

**JPEG** — для фотографий и сложных изображений с плавными переходами цветов. Поддерживает lossy-сжатие (с потерями). Не поддерживает прозрачность. Универсальная поддержка во всех браузерах.

**PNG** — для изображений, где важна чёткость, пиксель-арт, скриншоты, картинки с прозрачным фоном. Сжатие без потерь, файлы тяжелее JPEG.

**WebP** — современный формат от Google. Поддерживает и lossy, и lossless-сжатие, прозрачность, анимацию. Обычно даёт **на 25–35% меньший размер**, чем JPEG при том же визуальном качестве. Поддерживается всеми современными браузерами.

**AVIF** — ещё более современный формат на базе кодека AV1. Даёт лучшее сжатие, чем WebP, но медленнее кодируется и поддерживается не всеми старыми браузерами. Для продакшена используйте с fallback.

**SVG** — векторный формат. Идеален для иконок, логотипов, иллюстраций из геометрических форм. Масштабируется без потери качества, можно стилизовать через CSS, анимировать через CSS и JS. Не подходит для фотографий.

```html
<!-- Используем picture для progressive enhancement: AVIF → WebP → JPEG -->
<picture>
  <source srcset="photo.avif" type="image/avif">
  <source srcset="photo.webp" type="image/webp">
  <img src="photo.jpg" alt="Описание фото" width="800" height="600">
</picture>
```

### SVG для иконок

Вместо иконочных шрифтов (Font Awesome) или PNG-спрайтов используйте SVG:

```html
<!-- Инлайновый SVG — нет сетевого запроса, можно стилизовать -->
<svg width="24" height="24" viewBox="0 0 24 24" aria-hidden="true">
  <path d="M12 2L2 7l10 5 10-5-10-5z"/>
</svg>

<!-- SVG-спрайт через use — один запрос для всех иконок -->
<svg aria-hidden="true">
  <use href="/icons.svg#icon-home" />
</svg>
```

### Lazy loading

Изображения ниже линии сгиба (below the fold) не нужны пользователю немедленно. Используйте атрибут `loading="lazy"`:

```html
<img src="photo.webp" alt="..." loading="lazy" width="800" height="600">
```

Браузер сам решит, когда загрузить такое изображение — обычно когда пользователь приближается к нему при скролле.

Всегда указывайте `width` и `height` у изображений — это позволяет браузеру зарезервировать место и избежать **Cumulative Layout Shift (CLS)**.

## Веб-шрифты: форматы, font-display, FOUT и FOIT

Кастомные шрифты делают дизайн уникальным, но за это приходится платить временем загрузки.

### Формат WOFF2

Современный стандарт — **WOFF2**. Он использует алгоритм Brotli для сжатия и даёт примерно на **30% меньший размер**, чем WOFF. Поддерживается всеми актуальными браузерами. В 2024+ году достаточно подключать только WOFF2:

```css
@font-face {
  font-family: 'Inter';
  src: url('/fonts/inter-regular.woff2') format('woff2');
  font-weight: 400;
  font-style: normal;
  font-display: swap;
}
```

### Проблемы FOUT и FOIT

Когда браузер встречает текст, для которого нужен кастомный шрифт, но шрифт ещё не загрузился, он сталкивается с выбором:

- **FOIT (Flash of Invisible Text)** — скрывать текст, пока шрифт не загрузится. Пользователь видит пустое место. Плохо для UX и SEO.
- **FOUT (Flash of Unstyled Text)** — показывать текст в системном fallback-шрифте, а потом заменить на кастомный. Происходит «прыжок» текста.

Оба поведения можно контролировать через свойство **`font-display`**:

```css
@font-face {
  font-family: 'Inter';
  src: url('/fonts/inter.woff2') format('woff2');
  font-display: swap; /* показывает fallback сразу, заменяет на кастомный когда готов */
}
```

Значения `font-display`:

| Значение | Невидимый период | Замена | Когда использовать |
|---|---|---|---|
| `block` | 2–3 сек | есть | практически никогда |
| `swap` | нет | всегда | основной текст |
| `fallback` | ~100 мс | 3 сек | компромисс |
| `optional` | ~100 мс | нет | несущественные шрифты |
| `auto` | зависит от браузера | — | не рекомендуется |

**`font-display: swap`** — самый распространённый выбор. Пользователь сразу видит текст в системном шрифте, а когда кастомный загрузится — происходит замена.

**`font-display: optional`** — лучший выбор с точки зрения производительности. Браузер даёт шрифту только 100 мс, и если он не успел загрузиться — остаётся системный. Никакого layout shift, но шрифт может не применяться при медленном соединении.

### Минимизация влияния шрифтов на производительность

```html
<!-- Preconnect к Google Fonts — устанавливаем соединение заранее -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<!-- Preload критического шрифта — загружаем с высоким приоритетом -->
<link rel="preload" href="/fonts/inter-regular.woff2" as="font" type="font/woff2" crossorigin>
```

**Сабсеттинг** — подключайте только те символы, которые реально используются. Если сайт только на русском и английском языке, не нужен полный Unicode. Инструменты типа `pyftsubset` или онлайн-сервисы позволяют вырезать ненужные глифы и уменьшить файл шрифта в разы.

**Системные шрифты** — самая быстрая альтернатива, ноль сетевых запросов:

```css
body {
  font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, Oxygen,
    Ubuntu, Cantarell, sans-serif;
}
```
